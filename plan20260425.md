## 1. 목적

`test1`, `test2`, `test3` 계정과 대량 더미 데이터를 만들어 현재 백오피스의 실제 운영 화면을 부하와 다양한 상태 조건에서 점검한다.

이번 계획의 기준은 다음과 같다.

- 계정은 `test1`, `test2`, `test3` 3개
- 비밀번호는 모두 `0000`
- 세 계정 모두 같은 회사 `테스트커머스`
- 세 계정 모두 같은 메뉴 권한: 1 대시보드, 3 리뷰받기, 4 상품전체보기, 5 내보내기
- 세 계정 모두 `상품` 메뉴 권한 2번은 없음
- 한 계정당 상품 20개 이상 조건을 만족하도록 정확히 20개씩 생성
- 상품당 submissions는 10개 단위인 20개, 30개, 40개 중 하나로 생성
- 전체 생성량은 60개 상품, 1,740개 submissions 기준

점검 대상은 다음이다.

- 관리자 로그인
- 동일 권한 계정 3개의 메뉴 노출
- `상품` 메뉴 미노출 및 `/admin/product*` 직접 접근 제한
- 본인 데이터 / 같은 회사 데이터 토글
- 대시보드 KPI, 알림, 추이, 최근 활동, 회사 멤버 비교
- 리뷰받기 전체/진행중/완료/상세
- 상품전체보기 전체/상태별/필터/일괄 처리
- 구매자용 공개 리뷰받기 화면과 배정명 조회
- 내보내기 전체상품/내상품/일자별/상품별/입금일/상태별/신청자 명단
- 대량 행 렌더링, 필터링, 다운로드, 사진 있음/없음, 입금 지연, 완료 상품

## 3. 생성할 계정

비밀번호는 모두 현재 레거시 로그인 방식에 맞춰 평문 `0000`으로 넣는다.

| login_id | password | username | company | 목적 |
| --- | --- | --- | --- | --- |
| `test1` | `0000` | 테스트 관리자 1 | 테스트커머스 | 전체 권한, 기준 관리자 |
| `test2` | `0000` | 테스트 관리자 2 | 테스트커머스 | 같은 회사 범위 및 제한 권한 검증 |
| `test3` | `0000` | 테스트 관리자 2 | 테스트커머스 | 같은 회사 범위 및 제한 권한 검증 |

권한은 세 계정 모두 동일하게 준다.

| login_id | 허용 메뉴 |
| --- | --- |
| `test1` | 1 대시보드, 3 리뷰받기, 4 상품전체보기, 5 내보내기 |
| `test2` | 1 대시보드, 3 리뷰받기, 4 상품전체보기, 5 내보내기 |
| `test3` | 1 대시보드, 3 리뷰받기, 4 상품전체보기, 5 내보내기 |

권한 기준:

- 세 계정 모두 사이드바에 `대시보드`, `리뷰받기`, `상품전체보기`, `내보내기`만 표시된다.
- 세 계정 모두 `상품` 메뉴는 표시되지 않는다.
- 세 계정 모두 `/admin/product` 및 `/admin/product/specific/:productId` 직접 접근 시 권한 제한 흐름을 확인한다.
- 세 계정 모두 같은 회사이므로 회사 데이터 포함 토글 ON 시 3개 계정 데이터가 함께 보여야 한다.

예상 추가 권한 row 수:

- `admin_menu_permissions`: 12건
- 계정별 4건씩: menu_number 1, 3, 4, 5

## 4. 더미 데이터 설계 원칙

- 실제 개인정보는 사용하지 않고 전부 가짜 이름, 전화번호, 주소, 계좌번호를 쓴다.
- `products.id`, `submissions.id` 같은 identity 값은 하드코딩하지 않는다.
- 삽입 스크립트는 `manager_id`, `title`, `order_number` 같은 자연 키로 매핑한다.
- 재실행 가능하도록 `manager_id in ('test1', 'test2', 'test3')` 데이터만 FK 역순으로 삭제 후 다시 넣는 방식을 기본으로 한다.
- 삭제 순서: `evidence_photos` -> `submissions` -> `applications` -> `product_steps` -> `products` -> `admin_menu_permissions` -> `admins`
- 실제 운영 계정 `2sssg` 데이터는 절대 삭제하거나 수정하지 않는다.
- 날짜 기준은 2026-04-26 KST로 고정해 오늘/어제/최근 14일/입금 지연 케이스가 안정적으로 나오게 한다.
- 한 상품의 submissions 수는 반드시 20, 30, 40 중 하나로 한다.
- 한 계정의 상품 수는 정확히 20개로 한다.

## 7. 상품 필드 패턴

상품명은 화면에서 바로 식별되도록 테스트 목적과 key를 제목에 포함한다.

예시:

- `2026.04.26 / [T1-P01] 기본혼합 선크림`
- `2026.04.25 / [T1-P06] 리뷰대기 앰플`
- `2026.04.20 / [T1-P11] 완료많음 마스크팩`
- `2026.04.18 / [T1-P19] 입금지연 비누`
- `2026.04.26 / [T1-P20] 빈행일괄입력 샴푸`

필드 분포:

- `manager_id`: 각 계정의 `login_id`
- `company_name`: 기본 `테스트커머스`, 일부 null
- `option_name`: `단품`, `2개입`, `대용량`, null 혼합
- `review_type`: `텍스트`, `포토`, `영상`, null 혼합
- `review_fee`: 0, 1000, 1500, 2500, 3000, null 혼합
- `planned_depositor_name`: `0426테스트커머스`, `0425테스트커머스`, `0418테스트커머스`, null 혼합
- `deposit_date`: 2026-04-18부터 2026-05-10까지 분산
- `is_real_shipping`: true/false 혼합
- `created_at`: 2026-04-01, 2026-04-12, 2026-04-18, 2026-04-25, 2026-04-26로 분산

`product_steps`는 모든 상품에 1, 2, 3을 생성한다. 현재 세 계정에는 `상품` 메뉴 권한이 없지만, 데이터 계약 완전성과 이후 권한 변경 검증을 위해 단계 데이터는 채운다.

## 9. Submission 상태 패턴

상태는 현재 코드의 분류 기준을 따른다.

- 구매 섹션: `is_review_verified = false`
- 리뷰 섹션: `is_review_verified = true`, `is_deposit_verified = false`
- 완료 섹션: `is_review_verified = true`, `is_deposit_verified = true`

구매 섹션에는 다음 케이스를 섞는다.

- 구매정보가 모두 채워진 행
- `assign_name`만 있고 구매정보가 비어 있는 행
- `assign_name`이 null인 미배정 행
- `purchase_account`, `bank_account`, `amount` 일부 누락 행
- 구매 사진만 있는 행
- 사진이 전혀 없는 행
- P20의 40개 행은 구매정보 빈 행/일괄입력 검증용으로 유지

리뷰 섹션에는 다음 케이스를 섞는다.

- 리뷰 사진이 있는 행
- 리뷰 사진이 없는 행
- `created_at`이 2026-04-18 이전인 입금 지연 행
- `actual_depositor_name`과 `deposited_at`이 아직 null인 행
- P19의 리뷰 섹션 24개는 입금 지연 상태와 review 사진 필수 조건 검증에 집중

완료 섹션에는 다음 케이스를 섞는다.

- `deposited_at`이 2026-04-26인 오늘 입금 행
- `deposited_at`이 2026-04-25인 어제 입금 행
- `deposited_at`이 2026-04-01인 오래된 입금 행
- `actual_depositor_name`이 예정 입금자명과 같은 행
- `actual_depositor_name`이 예정 입금자명과 다른 행

공개 리뷰받기 조회용 `assign_name`은 세 계정 상품에 반복 사용한다.

- `김민지`: 여러 상품에 반복, 한 이름으로 다건 조회 검증
- `박서연`: 사진 업로드 가능 행 포함
- `이지훈`: 이미 리뷰완료되어 사진 수정이 잠기는 행 포함
- `최유나`: 완료 섹션 확인용
- `정도윤`: P20 빈 행 일괄입력 대상
- null: 미배정 알림 검증용

## 10. Evidence photo 패턴

`evidence_photos.photo_type`은 현재 코드에서 `purchase`, `review`를 사용한다.

사진 URL은 실제 S3 업로드 없이도 썸네일 동작을 확인할 수 있도록 placeholder URL을 쓴다.

예시:

- `https://placehold.co/640x480/png?text=purchase-t1-p01-001`
- `https://placehold.co/640x480/png?text=review-t1-p01-001`
- `https://placehold.co/640x480/png?text=review-t1-p19-missing-control`

분포:

- 구매 섹션 중 약 1/3: purchase 사진 1장
- 리뷰 섹션 전체: review 사진 1장 이상
- 완료 섹션 전체: review 사진 1장 이상
- 완료 섹션 대부분: purchase 1장 + review 1장 이상
- 구매 섹션은 review 사진이 있을 수도 있고 없을 수도 있으나, 현재 더미 데이터에서는 purchase 사진만 일부 보유한다.

초기 삽입 사진 row 수는 2,217건이었다. 이후 리뷰완료 또는 전체완료 처리된 submissions에는 모두 `photo_type = 'review'` 사진이 있어야 한다는 조건에 맞춰 누락된 review 사진 204건을 추가했다. 더미 데이터의 사진 row 수는 총 2,421건이다.

## 11. 화면별 검증 시나리오

### 로그인

- `test1 / 0000`: 로그인 성공, 대시보드/리뷰받기/상품전체보기/내보내기 표시
- `test2 / 0000`: 로그인 성공, 대시보드/리뷰받기/상품전체보기/내보내기 표시
- `test3 / 0000`: 로그인 성공, 대시보드/리뷰받기/상품전체보기/내보내기 표시

- 잘못된 비밀번호: 로그인 실패 메시지 표시

### 메뉴 권한

- 세 계정 모두 `/admin/product` 직접 접근 시 허용된 메뉴로 리다이렉트되는지 확인

- 세 계정 모두 `/admin/review-receive/*`, `/admin/product-overview/*`, `/admin/export/*` 접근 가능
- 권한 조회 중 본문이 먼저 렌더링되지 않는지 확인

### 회사 범위 토글

- `test1`에서 회사 데이터 포함 OFF: `test1` 상품 20개, submissions 580개 기준
- `test1`에서 회사 데이터 포함 ON: `test1` + `test2` + `test3` 상품 60개, submissions 1,740개 기준
- `test2`와 `test3`도 회사 데이터 포함 ON에서 동일하게 3개 계정 데이터가 보여야 한다.
- 세 계정 모두 company가 `테스트커머스`이므로 회사 정보 없음 메시지는 나오지 않아야 한다.

## 13. SQL 작성 시 주의사항

- 전체를 하나의 transaction으로 묶는다.
- identity PK는 직접 넣지 않는다.
- 재실행을 위해 dummy owner 범위를 `test1`, `test2`, `test3`으로 한정한다.
- `admin_menu_permissions`는 unique 계약인 `(admin_id, menu_number)` 기준으로 upsert한다.
- 권한은 세 계정 모두 menu_number 1, 3, 4, 5만 넣는다.
- menu_number 2 `상품` 권한은 넣지 않는다.
- `products.manager_id`와 권한의 `admin_id`는 반드시 먼저 생성된 `admins.login_id`를 참조한다.
- `evidence_photos`는 생성된 `submissions.id`를 매핑해서 넣는다.
- `created_at`은 timestamptz이므로 `2026-04-26T09:30:00+09:00` 형태를 사용한다.
- `deposit_date`, `deposited_at`은 date이므로 `YYYY-MM-DD` 형태를 사용한다.
- 상품당 submissions 수가 20, 30, 40 중 하나가 아니면 실패로 처리한다.
- 계정당 상품 수가 20개가 아니면 실패로 처리한다.

## 15. 완료 기준

더미 데이터 생성 작업은 다음 조건을 만족하면 완료로 본다.

- 세 계정 모두 `0000`으로 로그인 가능
- 세 계정 모두 같은 메뉴 권한이 화면에 반영됨
- 세 계정 모두 `상품` 메뉴가 보이지 않음
- 세 계정 모두 회사 데이터 포함 OFF에서는 본인 20개 상품만 보임
- 세 계정 모두 회사 데이터 포함 ON에서는 회사 전체 60개 상품이 보임
- 대시보드에 오늘/어제/입금지연/사진누락/회사멤버 비교 데이터가 노출됨

- 실제 운영 계정 `2sssg` 데이터가 변경되지 않음
