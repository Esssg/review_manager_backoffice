# 리뷰받기 상세 기능 확장 실행 계획

작성일: 2026-04-25  
대상 화면: `src/pages/admin/AdminReviewReceiveDetailPage.jsx`

## 1. 작업 목표

아래 요구사항을 `리뷰받기 상세` 화면에 반영한다.

1. `구매완료`, `리뷰완료`, `전체완료`를 각각 독립 섹션으로 유지한다.
2. 각 섹션 테이블 상단에 검색 입력칸을 추가하고, 입력 즉시 실시간 필터링되게 만든다.
3. 계좌 표시 형식을 `은행명 / 계좌번호 / 입금주`에서 `은행명 계좌번호 입금주`로 바꾼다.
4. `배정` 왼쪽에 순번 컬럼을 추가한다.
5. `구매완료` 섹션 상단에 `배정명 입력칸 + 일괄입력 텍스트칸 + 일괄입력하기 버튼`을 추가한다.
6. 일괄입력은 지정한 배정명에 맞는 행에 값을 채우고, 잘못된 입력에 대해 자동 보정 가능한 경우는 보정하고, 보정 불가능한 경우는 명확히 실패 처리한다.
7. `리뷰완료` 섹션 상단에 `일괄처리하기` 버튼을 추가하고, 모달에서 `입금일`, `실제입금자명`을 입력한 뒤 확인하면 해당 행들이 `전체완료`로 이동하게 만든다.

## 2. 현재 구조 기준 확인 사항

- 현재 화면은 이미 `구매완료`, `리뷰완료`, `전체완료` 3개 섹션으로 분리되어 있다.
- 주요 수정 대상은 아래 파일이다.
  - `src/pages/admin/AdminReviewReceiveDetailPage.jsx`
  - `src/services/reviewReceive.js`
  - `src/styles/admin-shell.css`
- 일괄입력용 파싱 로직은 재사용성과 테스트성을 위해 `src/utils/*`로 분리하는 방향이 적절하다.
- 현재 DB 컬럼으로 처리 가능한 범위는 아래와 같다.
  - `assign_name`
  - `order_number`
  - `buyer_name`
  - `recipient_name`
  - `contact`
  - `address`
  - `bank_name`
  - `bank_account`
  - `account_holder`
  - `amount`
  - `is_review_verified`
  - `is_deposit_verified`
  - `deposited_at`
  - `actual_depositor_name`
- 현재 요구사항만 보면 DB 스키마 변경은 필요 없어 보인다.

## 3. 구현 원칙

- 기존 `리뷰받기 상세` 구조를 유지하면서 필요한 상태와 UI만 확장한다.
- 검색, 파싱, 배치 처리 규칙처럼 순수 로직은 컴포넌트 밖 유틸로 분리한다.
- UI 추가 시 하늘색 톤과 기존 `dashboard-panel`, `admin-primary-button` 계열 스타일을 재사용한다.
- 에러는 콘솔에만 남기지 않고 화면에 명확한 메시지로 보여준다.
- 상태 전환은 즉시 보이되, DB 저장 실패 시 사용자에게 이유를 노출한다.

## 4. 요구사항별 세부 설계

### A. 섹션별 검색

- 각 섹션 헤더 아래 또는 같은 블록 안에 검색 입력칸을 추가한다.
- 검색은 입력 즉시 반영한다.
- 검색 대상은 섹션에서 사용자가 보는 텍스트 컬럼 전반으로 잡는다.
  - 순번
  - 배정
  - 주문번호
  - 구매자
  - 수취인
  - 연락처
  - 주소
  - 계좌
  - 금액
  - 입금자명(예정)
  - 입금일
  - 실제입금자명
- 검색 방식은 대소문자 무시 + 앞뒤 공백 제거 기준으로 처리한다.
- 검색은 각 섹션별 독립 상태로 관리한다.

### B. 계좌 표시 형식 변경

- 현재 `formatAccountParts()`와 `parseAccountInfo()`는 슬래시(`/`) 기준이므로 수정이 필요하다.
- 화면 표시 형식은 `국민은행 698902-01-368816 이혜미`처럼 공백 구분으로 통일한다.
- 신규 입력과 일괄입력도 같은 형식을 기준으로 처리한다.
- 내부 저장은 기존처럼 `bank_name`, `bank_account`, `account_holder` 컬럼을 유지한다.

### C. 순번 컬럼 추가

- `배정` 왼쪽에 `순번` 컬럼을 추가한다.
- 순번은 각 섹션의 현재 표시 순서를 기준으로 `1, 2, 3...`으로 렌더링한다.
- 검색 결과가 줄어든 경우에도 현재 보이는 목록 기준으로 다시 번호를 매긴다.

### D. 구매완료 섹션 일괄입력

- UI 구성
  - 입력칸 1: 배정명
  - 입력칸 2: 일괄입력 원문
  - 버튼: `일괄입력하기`
- 동작
  - 배정명 입력값으로 `구매완료` 섹션 안에서만 `assign_name`을 검색한다.
  - 검색 결과가 1건 이상이면 그 결과 목록을 현재 표시 순서대로 사용한다.
  - 일괄입력 텍스트칸은 여러 단락을 받을 수 있고, 단락은 줄바꿈(엔터) 기준으로 분리한다.
  - 각 단락을 순서대로 파싱해서 검색된 행에 앞에서부터 차례대로 매핑한다.
  - 검색 결과가 1건이고 입력 단락이 1건이면 그 1행에 반영한다.
  - 검색 결과가 여러 건이면 첫 번째 단락은 첫 번째 검색 결과, 두 번째 단락은 두 번째 검색 결과에 넣는다.
  - 검색 결과 수가 입력 단락 수보다 많으면 입력된 단락 수만큼만 앞에서부터 반영하고, 남는 행은 수정하지 않는다.
  - 검색 결과 수가 입력 단락 수보다 적으면 저장하지 않고 오류를 보여준다.
  - 반영 후 저장까지 한 번에 처리한다.
- 기본 기대 형식
  - `주문번호 / 구매자 / 수취인 / 연락처 / 주소 / 은행 계좌번호 입금주 / 금액`
- 자동 보정 허용 범위
  - 구분자 앞뒤 공백이 많은 경우
  - 연락처에 `-`가 포함된 경우
  - 금액에 `,` 또는 `원`이 포함된 경우
  - 계좌 필드 내부 공백이 여러 개인 경우
  - 문자열 앞뒤 공백이 있는 경우
- 명시적 실패 처리 대상
  - 배정명 누락
  - 배정명과 일치하는 구매완료 행이 없는 경우
  - 필수 조각 수가 맞지 않는 경우
  - 계좌 정보가 `은행/계좌번호/입금주`처럼 `/`로 잘못 분리된 경우
  - 필드 순서가 바뀌어 판단이 모호한 경우
  - 금액을 숫자로 해석할 수 없는 경우
  - 계좌 정보에서 은행/계좌번호/입금주 3요소를 안정적으로 분리할 수 없는 경우
  - 검색된 배정자 수보다 입력 단락 수가 많은 경우
- 예외 메시지 설계 방향
  - 어떤 필드가 왜 실패했는지 바로 보이게 작성
  - 자동 보정 가능한 경우는 조용히 처리
  - 자동 보정 불가 시 `입력 불가` 메시지로 종료
  - 검색된 배정자 수보다 입력 단락 수가 많을 때는 예시처럼 수량을 포함한 메시지로 종료
    - 예: `이혜미 배정자: 3명, 입력된 갯수: 4개`

### E. 리뷰완료 섹션 일괄처리 모달

- UI 구성
  - 리뷰완료 섹션 상단 버튼: `일괄처리하기`
  - 모달 필드: `입금일`, `실제입금자명`
  - 모달 하단 리스트: 현재 `리뷰완료` 섹션 행만 표시
- 기본 동작 가정
  - 모달에서 입력한 동일한 `입금일`, `실제입금자명`을 현재 리뷰완료 섹션의 모든 행에 일괄 적용한다.
  - 확인 시 각 행의 `is_deposit_verified`를 `true`로 저장한다.
  - 저장 성공 후 해당 행들은 `전체완료` 섹션으로 이동한다.
- 모달 검증
  - 리뷰완료 행이 0건이면 버튼 비활성 또는 눌러도 안내 메시지 처리
  - 입금일 누락 시 확인 불가
  - 실제입금자명 누락 시 확인 불가
  - 저장 중 중복 클릭 방지
  - 일부 실패 시 전체 실패로 볼지, 부분 성공으로 볼지 구현 전에 확정 필요

## 5. 승인 기반 단계별 실행 계획

아래 단계는 **각 단계 시작 전에 사용자 승인**을 받은 뒤 진행한다.

### 0단계. 계획 확정

- `plan.md` 검토 및 수정
- 모달 일괄처리 정책, 파싱 허용 범위, 메시지 문구 확정
- 승인 기준
  - 사용자가 `plan.md`를 수정하거나 승인 후 실행 지시

### 1단계. 섹션 UI 기본 확장

- 순번 컬럼 추가
- 계좌 표시 포맷 변경
- 각 섹션 검색 상태와 실시간 필터링 추가
- 검색/표시용 순수 유틸 분리 여부 반영
- 관련 스타일 보강
- 완료 후 확인
  - 섹션 3개가 정상 렌더링되는지
  - 검색 입력 시 행 필터링이 즉시 반영되는지
  - 순번이 보이는 목록 기준으로 다시 계산되는지

### 2단계. 구매완료 일괄입력 기능

- 구매완료 섹션 상단 배치 입력 UI 추가
- 파싱/보정/검증 유틸 분리
- 구매완료 섹션 내부 배정명 검색 후 다건 순차 매핑 처리
- 줄바꿈 기준 다중 단락 입력 처리
- 행 업데이트 + 저장 처리
- 실패 메시지와 성공 메시지 연결
- 완료 후 확인
  - 정상 형식 입력 성공
  - 공백/하이픈/금액 표기 자동 보정 성공
  - 잘못된 계좌 형식 실패
  - 잘못된 필드 순서 실패
  - 대상 배정명 없음 실패
  - 검색 결과보다 입력 단락이 많을 때 전체 삽입 차단
  - 검색 결과가 여러 건이면 첫 번째부터 순서대로 반영

### 3단계. 리뷰완료 일괄처리 모달

- 리뷰완료 섹션 버튼 추가
- 모달 UI 추가
- 리뷰완료 행 리스트 표시
- 입금일/실제입금자명 일괄 적용 저장
- 저장 성공 시 전체완료 섹션 이동
- 완료 후 확인
  - 리뷰완료 건이 있을 때만 정상 동작하는지
  - 확인 후 리뷰완료에서 사라지고 전체완료에 보이는지
  - 실패 시 데이터가 어중간하게 이동하지 않는지

### 4단계. 검증 및 마무리

- `npm run build`
- 필요 시 수동 점검 시나리오 정리
- 문서 갱신 필요 여부 검토
  - 구조가 크게 바뀌면 `docs/project_analysis.md` 갱신 검토
- 최종 변경 요약 작성

## 6. 구현 중 확정이 필요한 정책 포인트

아래 항목은 실행 전 사용자가 확인하거나 수정하면 구현 오차를 줄일 수 있다.

1. `리뷰완료 > 일괄처리하기`가 현재 리뷰완료 행 전체를 한 번에 처리하는 방식으로 맞는지
2. 일괄처리 저장 중 일부 행만 실패하면 전체 롤백이 필요한지, 성공한 행만 반영할지
3. 구매완료 일괄입력 시 배정명이 중복될 가능성이 있는지
4. 검색 대상에서 `입금자명(예정)`처럼 상품 공통값도 포함할지
5. 일괄입력 성공 후 입력칸을 자동 초기화할지

## 7. 검증 시나리오 초안

### 검색

- 구매완료 섹션에서 이름 검색
- 리뷰완료 섹션에서 주문번호 검색
- 전체완료 섹션에서 실제입금자명 검색
- 검색 결과 0건일 때 빈 상태 문구 확인

### 구매완료 일괄입력

- 정상 입력
  - `22222 / 이혜미 / 이혜미 / 010-2185-0826 / 경기도 용인시 ... / 국민은행 698902-01-368816 이혜미 / 17,000원`
- 다중 단락 입력
  - 같은 배정명 검색 결과가 3건이고 입력도 3줄이면 1행부터 3행까지 순서대로 반영
- 검색 결과가 더 많은 경우
  - 같은 배정명 검색 결과가 4건이고 입력이 2줄이면 첫 2행만 반영하고 나머지는 유지
- 자동 보정 입력
  - 앞뒤 공백 과다
  - 연락처 하이픈 포함
  - 금액에 쉼표, `원` 포함
  - 계좌 필드 내부 다중 공백
- 실패 입력
  - `국민은행/698902-01-368816/이혜미`
  - 필드 순서 변경
  - 금액 누락
  - 배정명 누락
  - 존재하지 않는 배정명
  - 검색된 배정자 3명인데 입력 단락 4개인 경우

### 리뷰완료 일괄처리

- 입금일, 실제입금자명 입력 후 일괄 완료
- 필드 하나라도 비어 있으면 확인 차단
- 저장 중 버튼 비활성
- 완료 후 전체완료 섹션 반영 확인
