# Review Manager Backoffice

## 목적

이 저장소는 `Vite + React + React Router + Supabase` 기반의 리뷰 운영용 백오피스 프런트엔드입니다.

이 문서는 Codex가 이 프로젝트에서 작업할 때 따를 기본 규칙입니다. 목표는 아래 4가지입니다.

- 기존 구조를 함부로 깨지 않고 유지보수 가능한 방식으로 확장할 것
- UI, 데이터, 라우팅, 문서를 함께 관리할 것
- 중복 구현보다 재사용과 분리를 우선할 것
- 결과물뿐 아니라 검증과 후속 정리까지 마칠 것

## 먼저 확인할 문서

작업 전에 아래 문서를 우선 확인합니다.

- `docs/project_analysis.md`
- DB 구조, 테이블 계약, 샘플 데이터와 관련 있으면 `docs/guide_db.md`
- Supabase 연결 자체를 다루면 `SUPABASE_SETUP.md`

기존 Cursor 규칙에서는 `docs/project_overview.md`를 참조하지만, 현재 이 저장소에는 해당 파일이 없습니다. 별도 개요 문서가 추가되기 전까지는 `docs/project_analysis.md`를 기준 개요 문서로 사용합니다.

## 핵심 작업 원칙

- 새 구현을 만들기 전에 기존 코드와 문서를 먼저 읽습니다.
- 작은 요구사항이면 현재 구조 안에서 최소 수정으로 해결합니다.
- 변경 범위가 커질수록 라우팅, 화면, 데이터 접근, 유틸 책임을 분리합니다.
- 사용자가 체감하는 동작 변경이 생기면 UI 상태, 에러 상태, 빈 상태까지 함께 점검합니다.
- 변경 후에는 최소한 빌드나 관련 검증을 실행해 회귀 여부를 확인합니다.

## 기존 구현 우선 재사용

새 파일이나 중복 로직을 만들기 전에 기존 코드를 확장하거나 재사용하는 방향을 우선합니다.

작업 순서는 아래를 따릅니다.

1. `docs/project_analysis.md`로 현재 구조와 책임 분리를 확인합니다.
2. 기존 구현에 같은 책임을 가진 코드가 있는지 먼저 찾습니다.
3. 가능하면 기존 코드를 수정하거나 확장해서 해결합니다.

현재 우선적으로 재사용을 검토할 위치:

- `src/App.jsx`: 현재 라우팅, 페이지, 화면 로직의 중심
- `src/lib/supabase.ts`: Supabase 클라이언트 설정
- `src/styles/login.css`: 현재 공용 스타일 레이어
- `scripts/check-supabase.mjs`: 연결 확인 스크립트

아래 조건을 만족할 때만 새 파일, 함수, 모듈을 추가합니다.

- 기존 코드로 요구사항을 깔끔하게 수용하기 어렵고
- 새 분리가 실제로 유지보수성과 책임 분리를 개선하며
- 중복 제거 또는 테스트 가능성 향상 같은 명확한 이점이 있는 경우

새 파일이나 추상화를 도입했다면, 왜 기존 재사용만으로는 부족했는지 작업 설명에 짧게 남깁니다.

## 파일 분리 기준

현재는 `src/App.jsx`에 로직이 집중되어 있으므로, 큰 작업에서는 아래 방향으로 점진적으로 분리합니다.

- 라우팅 정의: 최상위 라우트만 남기고 페이지 구현은 분리
- 페이지 컴포넌트: 화면 단위 책임만 가짐
- 데이터 접근: Supabase 쿼리와 변환 로직 분리
- 순수 유틸: 파싱, 포맷팅, 정렬, 검증 로직 분리
- 스타일: 화면 단위 또는 공통 토큰 기준으로 분리

권장 구조 예시:

- `src/pages/*`
- `src/components/*`
- `src/lib/*`
- `src/services/*`
- `src/utils/*`
- `src/styles/*`

단, 작은 수정이라면 억지로 파일을 쪼개지 않습니다. 분리는 실질적인 가독성과 책임 분리 개선이 있을 때만 합니다.

## React 개발 스타일

요즘 스타일의 React 작업 원칙은 아래를 따릅니다.

- 컴포넌트는 가능한 한 작고 읽기 쉽게 유지합니다.
- 파생 가능한 값은 상태로 중복 저장하지 않습니다.
- 이벤트 처리, 데이터 조회, 파싱, 렌더링 책임을 한 함수에 과도하게 섞지 않습니다.
- 부수효과는 `useEffect`에 몰아넣기보다, 실제 외부 동기화가 필요한 경우에만 사용합니다.
- 복잡한 리스트/폼/모달 로직은 별도 컴포넌트나 훅으로 분리합니다.
- `useMemo`, `useCallback`은 성능 병목이 명확할 때만 사용합니다.
- 로딩, 에러, 빈 상태를 생략하지 않습니다.

구체적 기준:

- 1개의 컴포넌트가 라우팅, 데이터 조회, 수정, 모달, 파싱, 스타일 제어까지 모두 담당하면 분리를 검토합니다.
- JSX 내부에서 장문의 조건문이나 데이터 변환이 반복되면 위로 끌어올리거나 유틸로 분리합니다.
- 폼은 입력값, 검증, 제출 상태를 명확히 구분합니다.

## 상태 관리 원칙

- 서버 원본 데이터와 화면 보조 상태를 분리합니다.
- 같은 데이터를 여러 상태로 중복 저장하지 않습니다.
- 낙관적 업데이트는 롤백 처리가 명확할 때만 사용합니다.
- 선택된 탭, 모달 열림 여부, 필터 같은 UI 상태는 화면 가까이에 둡니다.
- 여러 화면에서 재사용되는 도메인 상태가 생길 때만 상위로 끌어올립니다.

## 데이터 파싱과 유틸 규칙

- 문자열 파싱, 금액 변환, 정렬 기준 같은 순수 로직은 컴포넌트 밖으로 분리합니다.
- 입력 형식이 엄격한 경우에는 성공 케이스뿐 아니라 실패 메시지도 함께 설계합니다.
- 유틸 함수는 가능하면 순수 함수로 유지하고, DOM 접근이나 네트워크 호출을 넣지 않습니다.

예를 들어 `parseSubmissionText`, 금액 파싱, 정렬 로직은 재사용성과 테스트 가능성이 높으므로 분리 우선순위가 높습니다.

## Supabase 접근 규칙

Supabase는 단순히 연결만 되는 것보다, 어떤 책임을 어디에 두는지가 중요합니다.

- 새 쿼리를 추가할 때는 먼저 기존 쿼리 흐름과 중복 여부를 확인합니다.
- 쿼리는 필요한 컬럼만 선택하고, 의미 없는 `select("*")` 남발을 피합니다.
- 화면 컴포넌트 안에서 같은 쿼리를 여러 번 중복 작성하지 않습니다.
- 조회, 생성, 수정, 삭제 흐름은 가능하면 함수 단위로 분리합니다.
- 테이블명 예외 처리나 fallback 로직이 있으면 문서와 코드 양쪽에 일관되게 반영합니다.

현재 프로젝트에 맞춘 추가 원칙:

- `admins` 테이블 직접 조회 기반 로그인은 기존 레거시 흐름으로 취급합니다.
- 새 인증 기능을 추가할 때는 이 패턴을 확대 재생산하지 않습니다.
- 인증/권한 개선 작업이라면 `localStorage` 기반 식별자 저장보다 Supabase Auth 또는 서버 검증 기반 구조를 우선 검토합니다.
- 관리자 권한이 필요한 쓰기 작업은 권한 경계와 RLS 영향을 항상 함께 검토합니다.

## 데이터베이스 변경 규칙

Supabase 또는 MCP 도구를 사용해 `public` 스키마를 변경했다면, 같은 작업 안에서 반드시 `docs/guide_db.md`도 함께 갱신합니다.

적용 대상:

- 테이블 생성, 삭제, 이름 변경
- 컬럼 추가, 삭제, 이름 변경, 타입 변경, 기본값 변경, Null 허용 여부 변경
- Primary Key, Foreign Key, Unique, Index 변경
- `public` 스키마 구조에 영향을 주는 모든 마이그레이션

스키마 변경 후 반드시 수행할 작업:

1. 변경 적용 후 Supabase에서 최종 스키마를 다시 조회합니다.
2. 최종 상태에 맞게 `docs/guide_db.md`를 수정합니다.
3. 문서 안의 샘플 로우가 영향을 받으면 함께 최신화합니다.
4. 최종 응답에 `docs/guide_db.md`를 업데이트했다는 사실을 명시합니다.

문서화 범위는 기본적으로 사용자가 관리하는 `public` 스키마 객체에 한정합니다. `auth`, `storage` 같은 Supabase 내부 스키마는 별도 요청이 있을 때만 다룹니다.

## 라우팅 경계

관리자 영역과 공개 영역의 라우트를 명확히 분리합니다.

- 인증이 필요한 백오피스 페이지는 모두 `/admin/*` 아래에 둡니다.
- 공개 페이지나 구매자 페이지에는 `/admin/*` 네임스페이스를 사용하지 않습니다.
- 리다이렉트 추가 시 관리자 흐름과 공개 흐름을 섞지 않습니다.

예시:

- 관리자: `/admin/login`, `/admin`, `/admin/product`, `/admin/product/specific/:productId`
- 공개: `/join/:productId`, `/submit/:productId`, `/review/:submissionId`

## 관리자 메뉴 권한 규칙

관리자 메뉴는 프런트 하드코딩 노출이 아니라 DB 권한 기준으로 제어합니다.

- 관리자별 메뉴 노출 여부는 `public.admin_menu_permissions` 테이블을 기준으로 판단합니다.
- 권한 판별의 기준값은 `menu_number`입니다.
- `menu_label`은 DB에서 사람이 바로 식별하기 위한 표시용 컬럼으로 취급합니다.
- 특정 관리자 ID를 코드에 예외적으로 하드코딩해서 메뉴를 열어주지 않습니다.
- 사이드바에서 메뉴를 숨기는 것만으로 끝내지 않고, 해당 URL 직접 접근도 함께 제한합니다.

현재 메뉴 번호 계약:

- `1` = `대시보드` = `/admin`
- `2` = `상품` = `/admin/product`
- `/admin/product/specific/:productId` 같은 상세 경로도 `상품` 메뉴 권한에 포함해 취급합니다.
- `5` = `내보내기` = `/admin/export` (하위 `전체상품`, `내상품`, `일자별` 등 `/admin/export/*` 경로 포함)

구현 원칙:

- 새 관리자 메뉴를 추가할 때는 라우트만 만들지 말고 메뉴 번호, 라벨, 경로 매핑도 함께 갱신합니다.
- 메뉴 권한 경로 판별 로직과 사이드바 렌더링 로직은 같은 기준을 재사용합니다.
- `admin_menu_permissions` 운영 데이터 또는 초기 데이터도 함께 준비합니다.
- 메뉴 번호 계약이 바뀌면 관련 코드와 문서를 같은 작업 안에서 함께 수정합니다.

권한 처리 규칙:

- 권한 있는 메뉴만 왼쪽 관리자 메뉴에 표시합니다.
- 현재 경로가 권한 없는 메뉴라면 허용된 다른 관리자 메뉴로 리다이렉트하거나 권한 없음 상태를 표시합니다.
- 권한 조회 중에는 관리자 본문을 먼저 렌더링하지 말고, 로딩 또는 오류 상태를 명확히 보여줍니다.
- 권한 없음 상태를 만들었다면 빈 화면 대신 이유를 사용자에게 드러냅니다.

## UI와 스타일 규칙

이 프로젝트의 메인 컬러는 하늘색 계열입니다.

새 UI 작업 시 아래 원칙을 지킵니다.

- 기본 강조색과 주요 액션에는 하늘색 계열을 우선 사용합니다.
- 버튼, 포커스 링, 링크, 주요 배지 등 핵심 인터랙션 요소는 하늘색 톤으로 일관성을 유지합니다.
- 추가 색상 단계가 필요해도 `light sky`에서 `deep sky` 범위 안에서 확장합니다.

현재 코드 기준으로는 CSS 변수와 기존 색상 토큰을 우선 재사용합니다.

- `:root`의 색상 변수부터 확인합니다.
- 새 색을 추가하더라도 기존 하늘색 팔레트와 충돌하지 않게 만듭니다.
- 로그인 화면처럼 이미 시각 방향이 강한 UI는 전체 무드를 해치지 않게 확장합니다.

추가 UI 원칙:

- 반응형을 기본으로 고려합니다.
- 포커스 상태와 키보드 접근성을 유지합니다.
- 클릭 가능한 행, 버튼, 링크는 시각적으로 명확히 구분합니다.
- 로딩 중/비활성 상태를 숨기지 말고 드러냅니다.

## 타입과 데이터 계약

현재 저장소는 JS 중심이지만, 데이터 계약은 느슨하게 다루지 않습니다.

- Supabase 응답 구조를 사용할 때는 필요한 필드를 명확히 적습니다.
- 테이블 컬럼명, 상태값, 단계명 같은 도메인 문자열은 임의로 늘리지 않습니다.
- 동일한 데이터 구조가 반복되면 상수, 설명 주석, 타입 수준 문서화 중 하나로 정리합니다.
- TS 전환이나 타입 보강 작업은 점진적으로 진행하되, 혼합 상태를 더 혼란스럽게 만들지 않도록 합니다.

## 에러 처리와 사용자 피드백

- 실패 가능한 작업에는 사용자에게 보이는 에러 메시지를 제공합니다.
- 네트워크 오류, 입력 오류, 권한 오류를 가능하면 구분합니다.
- 삭제, 완료 처리, 단계 토글 같은 쓰기 작업은 성공/실패 후 UI 상태가 일관되게 남아야 합니다.
- 콘솔 로그만 남기고 화면에서는 침묵하는 상태를 만들지 않습니다.

## 공통 알림 모달 규칙

- 브라우저 기본 `alert`, `confirm`, `prompt`를 새로 추가하지 않습니다.
- 사용자 확인, 경고, 삭제 확인이 필요하면 `src/components/common/AppAlertDialog.jsx`를 사용합니다.
- 화면별로 알림 모달 디자인을 복제하지 말고, 버튼 라벨, 로딩 상태, 위험 동작 여부를 props로 전달합니다.
- 삭제 같은 위험 동작은 `variant="danger"`와 `admin-danger-button` 조합을 우선 사용합니다.
- 새 알림 디자인이 필요하면 먼저 `AppAlertDialog` 확장으로 해결할 수 있는지 검토합니다.

## 테스트와 검증

현재 저장소에는 테스트 인프라가 거의 없으므로, 작업 후 최소 검증을 반드시 수행합니다.

기본 검증:

- 프런트 코드 변경 시 `npm run build`
- Supabase 연결이나 환경변수 관련 작업 시 `npm run supabase:check`
- 라우팅 변경 시 주요 경로 진입 흐름 점검
- 폼 변경 시 성공, 실패, 빈 입력 케이스 점검

추가 원칙:

- 검증을 실행하지 못했다면 이유를 최종 응답에 명시합니다.
- 중요한 파싱 로직이나 제출 로직을 바꿨다면 최소한 수동 시나리오라도 확인합니다.
- 테스트 인프라를 새로 추가하는 경우, 사소한 UI 변경보다 재사용 가치가 큰 로직부터 검증합니다.

작업 중 생성한 임시 산출물은 종료 전에 삭제합니다.

예시:

- `tmp_*.md`, `scratch_*` 같은 임시 메모
- 1회성 쿼리 결과 덤프
- 임시 로그 파일이나 검증 출력 파일

저장소에는 유지할 필요가 있는 결과물만 남깁니다.
